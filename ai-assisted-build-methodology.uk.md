# Як ми побудували SOC compliance-inspector за один день за допомогою AI

**Аудиторія:** регулятори, аудитори, технічні керівники, яким доведеться відповісти «чи можеш ти привезти це в HQ і навчити?».
**Передумова:** команда з шести регуляторів, більшість з яких не використовували Claude Code до того ранку, побудувала робочий compliance-inspection tool ТА двомовний take-home-навчальний пакет приблизно за сім годин, під час воркшопу STEP UP 3! у Кракові (квітень 2026).
**Що це за документ:** чесний звіт про те, *як* це було зроблено — які промпти спрацювали, які — ні, де AI помилявся, і які техніки верифікації дозволили нам випустити інструмент, якому ми справді довіряємо.

---

## 1. Чесна преамбула — що AI вміє, а чого — ні

AI (зокрема Claude Code — термінальна agent-версія Claude) **надзвичайно добрий у**:

- Перетворення однореченневої специфікації компонента на робочу першу чернетку.
- Переклад з однієї мови на іншу (Python → Bash, English → Ukrainian, YAML → Python).
- Поясненні незнайомого коду, який ми підтягли з GitHub або від peer-команди.
- Ловленні boilerplate-помилок (забуті імпорти, неправильні type annotations).
- Підтриманні консистентності у десятку файлів (перейменування, рефакторинг, застосування однакового патерну скрізь).
- Написанні правдоподібних test fixtures.

Він **ще ненадійний у**:

- Прийнятті архітектурних рішень. На «X чи Y?» зазвичай повертає збалансований огляд, не рекомендацію на основі ваших конкретних обмежень.
- Зауваженні того, що компілюється, але *семантично* неправильне. Bearer-перевірка, яка завжди проходить, чудово компілюється.
- Дотриманні задекларованого scope. На промпт «build me a compliance dashboard» часто повернеться HTMX auto-refresh, якого ви не просили. Іноді бонус, іноді — scope creep.
- Повідомленні, коли не знає. Впевнено синтезує відповідь, що виглядає правильною. Ви все одно маєте верифікувати.

Ці обмеження — не теоретичні для нас. Ми зіштовхнулися з кожним під час воркшопу. Тому наступні секції — *переважно про верифікацію*, не про те, як краще промптувати.

## 2. Prompt-патерни, що спрацювали

Це патерни, не точні тексти — адаптуйте до свого завдання.

### 2.1 Спочатку скафолд, потім деталі

Поганий промпт: *«Build me a complete SOC compliance inspector.»*
(Повертається 400 рядків коду зі змішаними concerns, важко верифікувати.)

Кращий:

```
Start a FastAPI project in ~/app for a SOC compliance inspection tool. Create:
- main.py with /health and /compliance endpoints
- probes/__init__.py — registry pattern for probe modules
- storage/db.py — SQLite schema: findings(finding_id, probe_time, target_tool,
  check_id, check_label, expected, observed, status, severity, notes)
- schemas/envelope.py — shared Pydantic envelope model
- templates/compliance.html — Jinja HTMX dashboard
- scheduler.py — APScheduler, runs every 60 seconds
- systemd unit; requirements.txt.

Don't implement any probes yet — just the skeleton.
```

Отримуєте чистий скелет. Переглядаєте. Деплоїте. Потім додаєте по одному probe.

**Чому це працює:** розбиває задачу на ревʼюабельні шматки. Кожна відповідь Claude достатньо маленька, щоб прочитати повністю. Накопичувана впевненість.

### 2.2 По одній фічі за раз, з явним контрактом

```
In probes/, create auth_noheader.py. It registers a probe function with the
@register decorator. The function signature is:
    async def check(target_host: str, http_client: httpx.AsyncClient) -> dict

It should POST a minimal but valid envelope (no Authorization header) to
http://{target_host}/ingest and return a dict with:
    {check_id: "C-AUTH-001", expected: 401, observed: r.status_code,
     status: "PASS"|"FAIL"|"UNREACHABLE", severity: "high", notes: ...}

Exceptions should return status=UNREACHABLE with the exception text in notes.
```

Вказуйте **типи, форму повернення, поведінку при помилках**. AI заповнює тіло. Ви ревʼюєте.

**Чому працює:** AI не може відхилитися від вашого контракту, якщо контракт у промпті. Ви делегуєте імплементацію, не дизайн.

### 2.3 «Make it testable»

```
Write three pytest tests for this auth_noheader probe using respx to mock the
target. Test: (1) 401 → PASS, (2) 200 → FAIL, (3) connection error → UNREACHABLE.
```

AI пише тести швидше за людей. Тести також слугують документацією очікуваної поведінки — ви у майбутньому, читаючи їх, розумієте контракт функції за 30 секунд.

### 2.4 Explain-this

```
Explain what this Pydantic model does line by line, as if I have never used Pydantic.
[paste model]
```

10-секундний промпт, що може зекономити колезі 30 хв читання документації. Робіть це щоразу, коли успадковуєте код від Claude або peer-команди.

### 2.5 Двомовний переклад зі збереженням термінів

```
Translate this technical document to Ukrainian. Keep technical terms
(FastAPI, Pydantic, bearer token, rate limit, systemd, curl) in English.
Match the tone — factual, regulatory, auditor-oriented. Don't make it more casual.
```

AI за замовчуванням зберігає англомовну технічну лексику, якщо попросити. Без цього іноді перекладає «обмежувач швидкості» замість «rate limit».

### 2.6 Промпт критика

Коли Claude щось написав — **не просто приймайте**. Запустіть другий промпт:

```
Read the code I just asked you to write. What could go wrong in production?
What are the 3 edge cases you didn't handle?
```

Часто AI сам виявляє діри. Це значно дешевше, ніж знаходити їх в полі.

## 3. Промпти, що погано працювали

### 3.1 «Deploy the whole thing to production»

Якщо попросите Claude `systemctl restart` вашого сервісу або деплой через віддалений SSH — він часто це зробить. Але якщо не вдасться — ви не знаєте, в якому стані сервер. Тримайте деплої ручними і явними. Claude пише deploy-script, *ви* його запускаєте.

### 3.2 «Figure out why it's not working»

```
The /compliance endpoint is returning 500. Fix it.
```

Це крик про допомогу, що залишає Claude гадати. Краще:

```
Here are the last 30 lines of the gunicorn log. Here's the exception. Here's the
request that triggered it. Propose a minimal fix (not a rewrite).
```

Чим більше evidence ви вставляєте — тим кориснішою буде відповідь.

### 3.3 «Make it fast»

Claude додасть caching, async, threads, premature optimization, якщо дозволите. Не просіть «швидкого». Просіть «відповідає 15-секундному refresh-target» з вимірюваним критерієм.

### 3.4 «Match the existing code style»

У AI немає памʼяті між промптами, якщо ви не вставляєте знову. Просити «match the style of the rest of the project» без зразка — отримати впевнене, але неправильне. Вставте представницький файл і скажіть «match this structure».

## 4. Техніки верифікації — як ми уникали shipping broken code

Ми не довіряли AI сліпо. Кожен шматок, що йшов у production, верифікувався. Деякі з наших технік:

### 4.1 Трисекундний eyeball-тест

Для кожного файла, згенерованого AI, хтось читав зверху донизу перед деплоєм. Якщо ви не можете прочитати AI-згенерований код і передбачити його поведінку — розбивайте промпт на менші шматки або просіть AI пояснити.

### 4.2 Curl перед commit

Кожен HTTP endpoint, який ми випустили, верифікувався буквальним `curl` з іншого хоста. Не «код компілюється» — «endpoint повертає те, що має, коли його кличуть ззовні». Цей тест розрізняє `changeme` (placeholder, який AI автозаповнив) від справжнього токена.

### 4.3 Peer checks між командами

Команда 6 (Інспектор) проба Команду 3 (Аналітик) і отримала `401` відповідь на shared `SOC_PROTOCOL_TOKEN`. Це була compliance-знахідка — Analyst відійшов від протоколу. Це peer-review через трафік, не code review.

### 4.4 Читайте env, не лише код

Наш перший live-тест Команди 5 (Dispatcher) видався невдалим з `401`. Ми думали, що їхній app зламаний. Реальна причина: їхній `.env`-файл казав `SOC_PROTOCOL_TOKEN=changeme`, а app вантажив `~/.soc_env` зі справжнім токеном. Код був ок. Конфіг — ні.

**Урок:** коли bearer-check провалюється — читайте обидва кінці конфігу, не лише код.

### 4.5 Compile-level статичні перевірки

```bash
python3 -m py_compile **/*.py
```

Ловить syntax і import-помилки. Дешево; запускайте перед кожним деплоєм. AI іноді вводить circular imports при refactoring — compilation це ловить.

### 4.6 Grep літералів

Перед тим, як зробити наші репозиторії публічними, ми запустили:

```bash
grep -rInE "GhostTrace-0[1-6]!|sk-ant-|changeme|-----BEGIN.*PRIVATE KEY" .
```

Зловило десять hardcoded Lightsail-паролів у scratch-скриптах однієї команди — де AI «люб'язно» вставив пароль, бо команда раніше вклеїла його в промпт. **AI не видалить секрети, які одного разу побачив.** Ви маєте сканувати.

## 5. Моменти, де AI помилявся, і як ми це впіймали

Ми вели щоденник ([`methodology-journal.md`](methodology-journal.md)). Нижче — приклади з дня збірки, які ми зберегли як навчальні моменти.

### Момент 1 — Bearer token, що нічого не робив

**Контекст:** Команда 3 (Analyst) попросила Claude «add bearer auth on /ingest». Він додав. Код виглядав правильно. Тести пройшли.

**Що пішло не так:** app читав `os.getenv("SIEM_API_KEY", "changeme")` і порівнював вхідні токени зі значенням. Env var ніколи не був налаштований у production, тому app порівнював зі стрічкою `"changeme"`. Будь-який peer, що шле `Bearer <shared-soc-token>`, отримував `401`. Check був справжній; налаштоване значення — placeholder.

**Як це виявлено:** probe C-AUTH-002 Команди 6 повернула FAIL при надсиланні shared-токена. Саме для цього Команда Інспектора і існує.

**Урок:** AI із задоволенням напише bearer-check, який компілюється, проходить свої ж unit-тести, і ніколи реально не працює — бо налаштований секрет — placeholder. Peer-тестування це ловить. Unit-тестування саме по собі — ні.

### Момент 2 — Схема, що відійшла від контракту

**Контекст:** Команда 3 також модифікувала request-body модель для `/ingest` і додала обов'язкове поле `source`, якого в envelope протоколу не було. Peer-події без цього поля отримували `422`.

**Що пішло не так:** на прохання «add additional context» до моделі Claude додав обовʼязкове поле. AI не знав, що envelope — це контракт, спільний для пʼяти інших peer-інструментів. Ніхто йому не сказав.

**Як це виявлено:** integration testing показало, що peers проходять auth (`200` замість `401`), але одразу падають з `422`. Фікс тривіальний; інсайт — ні.

**Урок:** якщо ваш код імплементує контракт — **говоріть про це в кожному відповідному промпті**. Вставляйте документ контракту. Кажіть AI: «any modification to this model must not add required fields».

### Момент 3 — Admin-сторінка, що обслуговувала документи

**Контекст:** Команда 4 (Hunter) випустила admin-сторінку на порту 8001. Їхня probe C-ADMIN-001 пройшла — 401 без auth — виглядало добре. Потім хтось помітив, що `http://wic04:8001/` (голий root) повертає 200 OK з 30 КБ HTML.

**Що пішло не так:** команда зробила публічну «landing page» на `/`, яка описувала їхнє API, а захистила auth-ом лише `/admin`. Протокол казав «admin page requires auth» — не казав нічого про `/`. AI не мав жодної причини ставити auth на `/`.

**Як це виявлено:** ad-hoc curl на port root. Не жодним з 8 автоматизованих probes.

**Урок:** автоматизовані probes ловлять те, що ви задизайнили. Manual ad-hoc checks ловлять решту. Потрібні обидва.

## 6. Коли зупинитися використовувати AI і думати самостійно

Є моменти у такій роботі, де AI сповільнить вас, а не прискорить:

1. **Коли scope незрозумілий.** «Should this tool do X or Y?» — це рішення, не задача. AI дасть reasoning для обох; ви маєте обрати. Не дозволяйте схильності AI до збалансованих оглядів завадити вам вирішити.

2. **Коли ви дебажите мовчки-зламану інтеграцію.** Якщо curl повертає 401 і ви не розумієте чому — дивитися на код швидше, ніж промптувати. Grep env-змінних running-процесу. Читайте живий HTTP-response з `-i`. Людська спостережність тут досі виграє.

3. **Коли AI зациклюється.** Якщо після трьох промптів він досі не робить те, що вам треба — зупиніться. Перепишіть промпт з нуля, з більшим контекстом. Або перейдіть на самостійне написання цього шматка.

4. **Коли security posture — це суть.** «Is this auth check actually secure?» — людське питання. AI допоможе вам reasoning, але відповідь ви власнете. Наш incident з bearer-check Команди 3 — не рідкісний випадок; це типовий.

5. **Коли ви вирішуєте, що shippити в production.** AI чудовий для чернеток. Він поганий суддя у «this is good enough?» — бо не бачить ваш operational context. Ви вирішуєте.

## 8. Принесення цього в HQ

Запропонована навчальна послідовність для колег:

1. **Demo-сесія (30 хв).** Покажіть Claude Code в дії. Зробіть тривіальне завдання наживо (скафолд простого FastAPI-app). Неполіровано; нехай побачать, як ви ітеруєте.
2. **Кероване вправу (2 години).** Кожен колега обирає маленьку задачу — додати нову compliance-перевірку, перекласти документ, написати тест. Вони ведуть, ви коучите.
3. **Перше реальне використання (1–2 тижні пізніше).** Кожен колега використовує Claude Code для одної реальної задачі в actual workflow, веде щоденник (так само, як ми), повертається з рапортом, що спрацювало, а що ні.
4. **Group-ретроспектива (1 година).** Усі діляться journal-ами. Визначаємо патерни, що зʼявляються. Промоутимо найкращі промпти в team-shared-бібліотеку.

Це шеститижневий шлях від «ніколи не чув про Claude Code» до «використовую щодня для конкретних, вимірюваних задач». Пришвидшення за цей темп зазвичай створює або resistance, або over-reliance. Обидва — погані.

## 9. Те, що ми не пробували, але могло б допомогти

Для чесності. Якщо робитимете цю вправу ще раз:

- **Pre-scaffold протокол-документ як промпт.** Могли б змусити кожну команду попередньо завантажити soc-protocol-спец в першу Claude-сесію як контекст. Можливо, уникнули б schema-drift Команди 3.
- **Спільні prompt-шаблони на команду.** Ми дали кожній команді англомовні промпти inline в їхньому плані. Якби централізували prompt-бібліотеку (для `/ingest`-імплементацій, admin-auth тощо) — дивергенція була б меншою.
- **Continuous compliance протягом дня.** Команда 6 виробила знахідки наприкінці. Якби вони робили їх щогодини — інші команди мали б час виправити.
