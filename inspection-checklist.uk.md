# Чек-лист інспекції SOC на відповідність

**Джерело:** Команда 6 — Інспектор, STEP UP 3! Kraków 2026
**Призначення:** повторно-використовуваний чек-лист, який регуляторний інспектор може взяти в будь-який Security Operations Center для перевірки відповідності протоколу. Вісім перевірок, кожна — з командою ручної верифікації, очікуваним результатом і властивістю безпеки, яку вона підтверджує.

Використовуйте в описаному вигляді для first-pass аудиту. Будь-який `FAIL` — це невідповідність, яку оператор SOC має усунути до закриття аудиту.

---

## Як користуватися чек-листом

1. Встановіть scope з оператором SOC — які інструменти / хости / порти входять в область? Отримайте від оператора декларацію того, які endpoint-и мають бути публічними, а які — захищеними.
2. Для кожної перевірки нижче — виконуйте вказану команду з робочої станції **поза** trust-boundary SOC. Не запускайте перевірки зсередини мережі SOC — ви маєте перевірити те, що бачить зовнішній зловмисник.
3. Занотуйте `expected` vs. `observed` для кожного прогону. Будь-яке відхилення — це finding. Не судіть про severity в полі. Все записуйте; triage робіть пізніше.
4. Після виконання всіх перевірок — пройдіться по знахідках з оператором. Порядок перевірок нижче — свідомий: пізніші припускають, що попередні пройшли.

Для прозорості аудиту — ваші перевірки та частота probing мають бути задекларовані оператору SOC до початку. Підтримуваний флудинг target-у неприпустимий.

## Вісім перевірок

### C-LIVE-001 — Liveness сервісу

**Що тестує:** що інструмент працює і досяжний.

**Ручна верифікація:**
```
curl -i --max-time 3 http://<host>:<port>/health
```

**Очікується:** HTTP 200, тіло — JSON, що містить щонайменше `"status": "ok"`.

**Наслідки провалу:** якщо інструмент не працює — жодна інша перевірка не має сенсу. Позначте як `UNREACHABLE` і рухайтеся далі; повторіть усі перевірки, коли інструмент повернеться в робочий стан.

---

### C-AUTH-001 — Неавторизований write відмовлений

**Що тестує:** endpoint `/ingest` SOC відмовляє у запитах, де зовсім немає заголовка `Authorization`.

**Ручна верифікація:**
```
curl -i -X POST --max-time 5 \
  http://<host>:<port>/ingest \
  -H "Content-Type: application/json" \
  -d '{"schema_version":"1.0","event_id":"test-0001","event_type":"telemetry","timestamp":"2026-04-23T10:00:00Z","producer":"inspector","severity":"low"}'
```

**Очікується:** HTTP **401**.

**Наслідки провалу:** якщо повертається `200`/`202` — будь-який користувач з інтернету може інжектити довільні події в pipeline SOC. Це пряме порушення задекларованого протоколу §7.1 (bearer-token обовʼязковий на `/ingest`).

---

### C-AUTH-002 — Неправильний bearer token відмовлений

**Що тестує:** endpoint валідує значення токена, не просто його наявність.

**Ручна верифікація:**
```
curl -i -X POST --max-time 5 \
  http://<host>:<port>/ingest \
  -H "Content-Type: application/json" \
  -H "Authorization: Bearer WRONG_TOKEN_XYZ" \
  -d '{"schema_version":"1.0","event_id":"test-0002","event_type":"telemetry","timestamp":"2026-04-23T10:00:00Z","producer":"inspector","severity":"low"}'
```

**Очікується:** HTTP **401**.

**Наслідки провалу:** якщо неправильний токен приймається — перевірка bearer лише косметична. Атакуючі часто зондують надсиланням `Bearer anything`; незвідане прийняття тут еквівалентне відсутності auth.

---

### C-SCHEMA-001 — Non-JSON тіло відмовлене

**Що тестує:** input-валідація на `/ingest`. Endpoint не повинен падати, давати 500 або приймати не-JSON.

**Ручна верифікація:**
```
curl -i -X POST --max-time 5 \
  http://<host>:<port>/ingest \
  -H "Authorization: Bearer <valid-token>" \
  -H "Content-Type: text/plain" \
  -d 'this is not json at all'
```

**Очікується:** HTTP **400**.

**Наслідки провалу:** `500` тут свідчить про необроблене виняткове поле — проблема надійності та вектор DoS. `200`/`202` — endpoint мовчки відкидає malformed input, не записуючи security event.

---

### C-SCHEMA-002 — Відсутні обовʼязкові поля envelope відмовлені

**Що тестує:** endpoint відмовляє події, де відсутні обовʼязкові поля envelope (`schema_version`, `event_id`, `event_type`, `timestamp`, `producer`, `severity`).

**Ручна верифікація:**
```
curl -i -X POST --max-time 5 \
  http://<host>:<port>/ingest \
  -H "Authorization: Bearer <valid-token>" \
  -H "Content-Type: application/json" \
  -d '{"event_type":"telemetry"}'
```

**Очікується:** HTTP **400**.

**Наслідки провалу:** якщо недоспецифіковані події приймаються — downstream кореляція в SIEM та SOAR не може розраховувати на присутність жодного поля. Контракт з downstream-інструментами ламається мовчки.

---

### C-RATE-001 — Rate limit спрацьовує

**Що тестує:** тривалі high-rate POST-и до `/ingest` (з валідним токеном і валідним payload-ом) рано чи пізно спрацьовують явний rate-limit.

**Ручна верифікація:**
```
for i in $(seq 1 200); do
  curl -sS -o /dev/null -w "%{http_code}\n" -X POST \
    http://<host>:<port>/ingest \
    -H "Authorization: Bearer <valid-token>" \
    -H "Content-Type: application/json" \
    -d "{\"schema_version\":\"1.0\",\"event_id\":\"burst-$i\",\"event_type\":\"telemetry\",\"timestamp\":\"2026-04-23T10:00:00Z\",\"producer\":\"inspector\",\"severity\":\"low\"}"
done | sort | uniq -c
```

**Очікується:** щонайменше одна відповідь **429** у розподілі кодів.

**Наслідки провалу:** якщо кожен запит повертає 202 під час флуду — у інструмента немає rate-limit, і легітимний peer (або скомпрометований) може насититу pipeline.

**Примітка аудитора:** запускайте цю перевірку щонайбільше **раз на годину** на target. Повторний швидкий прогін перестає бути probe і стає DoS-тестом; тримайтесь з правильного боку цієї межі.

---

### C-ADMIN-001 — Admin-сторінка відмовляє неавторизованим

**Що тестує:** admin-сторінка на порту 8001 вимагає автентифікації.

**Ручна верифікація:**
```
curl -i --max-time 3 http://<host>:8001/admin
```

**Очікується:** HTTP **401**.

**Наслідки провалу:** поверхні конфігурації оператора або аудиту, що витікають без auth — класична мі́сконфігурація SOC. Атакуючі постійно перебирають `/admin`, `/dashboard`, `/console`.

---

### C-ISOLATION-001 — Admin-роути не віддаються на app-порту

**Що тестує:** admin-роути живуть **лише** на порту 8001. Публічний app-порт (8000) не має віддавати жодний admin-роут.

**Ручна верифікація:**
```
curl -i --max-time 3 http://<host>:8000/admin
```

**Очікується:** HTTP **404**.

**Наслідки провалу:** `401` або `200` тут свідчить — admin-застосунок прив'язаний до публічного порту. Навіть якщо він credential-protected, будь-яка CVE в admin-стеку стає інтернет-exposed. Конвенція протоколу про розподіл портів існує саме для того, щоб зменшити цей blast radius.

## Оформлення знахідки

Для кожного `FAIL` записуйте:

| Поле | Приклад |
| ---- | ------- |
| Check ID | `C-AUTH-001` |
| Target (host + port) | `wic03.sanctumsec.com:8000` |
| Time of observation (UTC) | `2026-04-23T14:02:14Z` |
| Очікувалось | `401` |
| Отримано | `202` |
| Команда відтворення | (вставте точний `curl`-виклик) |
| Evidence | (response headers + перші 500 байт тіла) |
| Рекомендована ремедіація | коротко, дієво — не пишіть архітектурні поради тут |

Тримайте знахідки фактологічними. Поради щодо ремедіації йдуть в окрему секцію або follow-up engagement.

## Що цей чек-лист **не** покриває

- Anti-abuse контроли за межами rate-limit (reputation, geo-blocking, captcha)
- Secret hygiene (аудит коду на hardcoded tokens/passwords — робіть окремо проти репозиторію)
- Operational телеметрія (логи, метрики, audit trails)
- Incident-response runbook-и
- Availability / DR-постава

Це глибші engagement-и. Цей чек-лист — **30-хвилинна external-perimeter compliance-перевірка**, яку ви можете запустити проти будь-якого SOC, що задекларував імплементацію контракту `soc-protocol`.
